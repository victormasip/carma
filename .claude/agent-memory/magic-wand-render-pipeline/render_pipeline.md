The "Magic Wand" / Theme clone pipeline.

**Current approach (2026-06-02 PIVOT — supersedes BOTH the LLM-reconstruction era AND the headless-browser+shadow-transplant era):** RAW HTML INJECTION. User manually verified this gives a 1:1 clone; the two prior approaches were rejected as unreliable/inaccurate.

How it works (UPDATED 2026-06-03 — "Top/Bottom sandwich", see decision-log entry 36):
1. Static fetch of the target. Extract the client's REAL `<head>` inner HTML (stylesheet/style/font links + preconnect/preload + scripts), absolutized to the origin; STRIP `<title>`/`<meta>`/`<link rel=canonical>`/`<base>` AND now drop SPA-bootstrap/hydration/service-worker/redirect scripts + trackers (`shouldDropScript` in headerFooter.ts) so the clone can't blank.
2. **`splitPageChrome(html, base)` (NEW `src/lib/scrape/pageSplit.ts`, parse5)** carves the page into a SANDWICH around its main content: `extracted_header` = the "Top" (everything before main — opening wrappers + header), `extracted_footer` = the "Bottom" (everything after — footer + the matching wrapper CLOSERS + late scripts), `extracted_body_attrs` = the `<body>`'s class/style/data-*. parse5 repairs malformed markup first (browser-accurate), so a missing `</div>` can't swallow the page. Strategies: `main` (carve the main-content node) › `header-footer` (LCA carve) › `region-fallback` (balanced single regions, never worse than before) › `none`. NOT `<header>`/`<footer>` outerHTML in isolation anymore — that was problem #2/#4.
3. Render doc = OUR `<head>` SEO + `[extracted_head]` + base reset (BEFORE client head, so `body{}` wins) + host box-guard (AFTER client head), then `<body{extracted_body_attrs}>` with the body assembled by **server-side stitching** (`stitchChrome` in theme.ts): `normalizeFragment(Top + '<!--CARMA_BLOG_SLOT-->' + Bottom)` parses the pair with parse5 → ONE BALANCED shell with the slot in place → splice the (balanced) Shadow-DOM blog host into the slot. **INVARIANT (2026-06-03): we NEVER ship unbalanced "open in header / close in footer" halves to the browser — the document is made well-formed server-side first.** This killed the "browser mis-parses the split → broken UI" bug. `regionHtml` returns each region's HTML unbalanced; `stitchChrome` balances the whole. (`tagBalanceDelta`/`buildChromeSandwich` were removed.) The source's wrappers re-wrap our blog (correct layout) and `body.theme-dark{…}` rules match (correct background).
4. **Legibility (CTO mandate):** the blog is FORCED readable-light. `tokensOf` runs `ensureReadableTokens` — a dark/low-contrast scraped palette falls back to light surface defaults (brand accent/fonts kept); `color-scheme:light` is set on the blog `:host`/`.carma-root`/embed fragment so inherited system/client dark mode can't reach it. Only the BLOG is forced light — the chrome keeps the client's look. The site Name is NEVER overwritten from scraped metadata (`SiteDetailClient.handleCaptureSuccess` no longer calls `updateSiteName`).

**Why this satisfies CSS isolation (the load-bearing insight):** the client's CSS is GLOBAL in the light DOM, so the injected header/footer are pixel-identical to the source. The blog lives behind a Shadow DOM boundary, so the client's global CSS (resets, `body{}`, `*{}`) cannot pierce in and break our blog, and our blog CSS cannot leak out onto the chrome. No iframe anywhere. We do NOT rewrite URLs inside external CSS files — linking them at their absolute origin URL makes their own relative font/image URLs resolve automatically (this is the key to the 1:1 result).

Key files (post-pivot):
- `src/lib/scrape/pageSplit.ts` (NEW, parse5) — THE carve module: `splitPageChrome` + `MAIN_SELECTORS` + a minimal CSS matcher. Pure/testable (relative imports only). This is where the Top/Bottom split, malformed-HTML repair, body-attr capture and script sanitisation live.
- `src/lib/scrape/headerFooter.ts` — head extraction + shared primitives. `buildExtractedHead` (filters head: keeps stylesheet/style/font/hint links + safe scripts; strips title/meta/canonical/base + DROPS dangerous/tracker scripts), `shouldDropScript` (anti-blank-page denylist), `tagBalanceDelta` (sandwich-vs-legacy probe), `absolutise`/`absolutiseMarkup`, `normalizeFragment`, plus the legacy `rawRegionHtml`/`findRegionEl`/`HEADER_SELECTORS`/`FOOTER_SELECTORS` (still exported; `pageSplit` reuses the selector tables). Imports `./clientCss` by RELATIVE path so it runs under plain Node for the .mjs tests.
- **Headless verification** that works here: `node --experimental-strip-types <test>.mjs` + a resolve hook that maps `@/`→`src/` and appends `.ts` to extensionless relative imports (parse5/node-html-parser resolve as bare specifiers). Lets a .mjs import the REAL `pageSplit`/`theme` and assert on synthetic broken HTML + a live fetch. 44 asserts green this session.
- `src/app/api/theme/analyze/route.ts` — SSE capture; imports the helpers above. `extracted_head`/`extracted_header`/`extracted_footer` are RAW absolutised HTML strings (NOT JSON). Keeps `collectHeadStyles`/`extractScripts`/`fetchAllCss` — CSS is fetched ONLY to feed `extractTokens`; the OUTPUT links to CSS, never inlines it.
- `src/lib/render/theme.ts` — `buildHead` injects `sanitizeInjectedHead(extracted_head)` + `buildPageBaseCss` (the blog template CSS now lives INSIDE the shadow host via `renderBlogHost`). `parseRegion` → `{kind:'raw'|'scoped'}`: raw HTML → light DOM verbatim (`<div class="carma-chrome…" data-carma-chrome>`); starter-template `{html,css}` → `scopeChromeCss`; legacy `{...,mode:'shadow'}` → degrade to raw `.html`. `runtimeScript()` (DSD polyfill) ALWAYS emitted (blog is in a shadow). `listingBlogInner`/`articleBlogInner` are the blog markup (reused by full page AND the embed fragment, which ships blog-only). Article JSON-LD carries `articleBody` (crawlable copy, since prose is in shadow).
- `src/lib/scrape/clientCss.ts` — KEPT but TRIMMED to URL helpers only (`absolutiseUrl`/`absolutiseCssUrls`/`splitImports`); the old `scopeClientCss` shadow-remap was deleted.
- `src/lib/render/scopeCss.ts` (`scopeChromeCss`) + `src/lib/render/templates.ts` + `src/app/api/theme/template-preview/route.ts` + `SiteOnboarding.tsx` — KEPT. The "start from a starter template" onboarding path is a DISTINCT feature; templates store `{html,css}` and render self-contained via scopeChromeCss. Do NOT delete.
- Theme Studio UI: `VisualChromeEditor.tsx` was rewritten from click-to-edit (which assumed per-region `{html,css}`) to a RAW HTML source editor per region + a collapsible `<head>` editor (raw chrome has no per-region CSS to inspect). `NavEditor.tsx`/`navEdit.ts` still work (they edit the link set; `parseChromeValue`/`serializeChromeValue` now round-trip raw→raw vs JSON→JSON via a `raw` flag). `ThemeStudioContext` exposes `extractedHead`/`setExtractedHead`. `actions/theme.ts` `translateChrome` handles raw vs JSON.

**DELETED in the pivot (do NOT reintroduce):** `src/lib/scrape/browser.ts` (capturePage/computed capture), `src/lib/render/shadowChrome.ts` (buildShadowChrome/CHROME_RUNTIME/sanitizeChromeCss), `src/lib/scrape/chromeCss.ts` (extractChromeCss), `src/lib/render/llmChrome.ts`, `src/app/api/theme/preview/route.ts` (was DEAD — no callers). `BROWSER_RENDER_URL` env + computed-style tiers are gone. `clientCss.ts`'s `scopeClientCss` deleted (rest of the file KEPT).

The blog ALWAYS renders inside a Shadow DOM; injected client head/header/footer live in the light DOM. `scopeChromeCss` still scopes the starter-template chrome. NO iframe for the render.
